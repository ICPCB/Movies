================================================================================
CINEMATCH — ENGLISH-LANGUAGE MOVIE RECOMMENDER
Project Explanation for the Reviewer
================================================================================

This document explains, in English, what the CineMatch project does, how it is
built, the design decisions behind the current state, and how to read the
benchmark numbers.  It is meant to stand on its own — a reviewer should be able
to read this file and understand the system without opening the source code.

A short presentation script (~2-3 minutes spoken) appears in Section 11.

1. PROJECT GOAL AND SCOPE
================================================================================

CineMatch is an English-language natural-language movie recommendation system.
A user types a plain-English movie request into a Gradio interface — for
example, "a dream heist movie where people enter dreams to steal secrets" — and
receives a ranked list of relevant movies, each with a short explanation
grounded in the movie's actual TMDB metadata.

IN SCOPE
    - English plain-language movie search.
    - Dense semantic retrieval with BAAI/bge-m3.
    - BM25 keyword retrieval over a field-aware index.
    - Reciprocal Rank Fusion (RRF) of the two retrievers.
    - Cross-encoder reranking with BAAI/bge-reranker-v2-m3.
    - Metadata-grounded LLM query expansion, HyDE, and explanations.
    - Robust deduplication on a stable (title, year) key.
    - Repeatable benchmark with 10 expected-title cases.

OUT OF SCOPE (current build)
    - Vietnamese or multilingual query support.
    - Personalised recommendations based on viewing history.
    - User accounts, ratings, watch history, or long-term memory.
    - Any claim that the system uses viewing history.
    - Automatically rebuilding the vector store from the running app.

The current scope is English-query retrieval over English TMDB metadata.
Movies may have any original language when the metadata fields used by the
index are available in English. Multilingual query translation is listed in
Section 12 as future work.

2. DATASET
================================================================================

Source        TMDB movie metadata, post-cleaning.
Rows kept     Defined by DATASET_ROW_COUNT in src/config.py; verify live CSV
              drift with scripts/print_dataset_stats.py.

Filter chain in `01.clean_data.py`:
    - Drop rows where `status` is not "Released".
    - Drop rows with missing overview / title / release date.
    - Keep all original languages when English TMDB metadata is available.
    - Drop rows with `vote_count < 50` to suppress obscure / low-signal items.
    - Deduplicate on (lower(title), year).

Each surviving row carries:
    id, title, original_title, overview, tagline, keywords (cleaned),
    genres + genres_clean, year, vote_average, vote_count, popularity,
    release_date, poster_path, runtime, original_language, plus a
    pre-rendered `document` field used by the embedder.

The vector store is ChromaDB.  Every row is embedded with `BAAI/bge-m3` and
stored under the stable id `tmdb_{id}`.  Stored metadata includes
`keywords`, `tagline`, `vote_count`, `year`, `genres_clean`, `popularity`,
`overview`, and friends, so downstream components can read what they need
without re-loading the CSV.

Rebuild step is explicit, never automatic:

    Remove-Item -Recurse -Force data/chroma_bgem3
    Remove-Item -Force data/movies_clean.csv
    python 01.clean_data.py
    python "02. Embed_BGEM3.py"

3. ARCHITECTURE OVERVIEW
================================================================================

The project keeps a strict separation between layers — `app.py` is only the UI
and pipeline dispatcher; all retrieval logic lives under `src/retrieval/`, all
pipeline orchestration under `src/pipelines/`, all LLM client and prompt code
under `src/llm/`.  This separation is enforced as a hard rule in `AGENTS.md`.

    app.py                          Gradio UI + dispatcher only

    src/
      config.py                     Shared constants and knobs
      models.py                     Lazy singleton model loaders

      retrieval/
        semantic.py                 BGE-M3 + ChromaDB semantic search
        bm25.py                     Field-aware BM25 retrieval
        fusion.py                   Reciprocal Rank Fusion
        reranker.py                 Cross-encoder reranking
        filters.py                  Natural-language filters
        query_processor.py          Query normalization

      pipelines/
        basic.py                    Semantic-only
        advanced.py                 Semantic + LLM expansion + HyDE + rerank + explain
        hybrid.py                   Semantic + BM25 + RRF + rerank + explain

      llm/
        langchain_ollama.py         LangChain + Ollama client, evidence verifier
        prompts.py                  Expansion, HyDE, and explanation prompts

      utils/
        dedup.py                    Stable movie_key + deduplication
        debug.py                    DEBUG_RETRIEVAL-gated stage timings

    scripts/
      quality_smoke_test.py         10-case English benchmark with expected titles
      _adhoc_benchmark.py           Lightweight retrieval-only benchmark
      _adhoc_explain_test.py        Verifies evidence-grounded explanations

The rule "do not put LLM calls in `src/retrieval/`" is load-bearing.  If a
retrieval module starts calling an LLM at request time, retrieval becomes
non-deterministic and slow, which is unacceptable for the benchmark and for
day-to-day app responsiveness.  LLM work happens in pipelines, not retrievers.

4. RETRIEVAL COMPONENTS
================================================================================

4.1  SEMANTIC RETRIEVAL
    Model:        BAAI/bge-m3 (multilingual dense embedder; English metadata here)
    Store:        ChromaDB local persistent client (data/chroma_bgem3)
    Top-K:        CANDIDATE_POOL (default 300)

    On a query we embed the query string with bge-m3 and ask ChromaDB for the
    nearest documents by inner-product similarity.  The output is a list of
    dicts with the movie's metadata plus `semantic_score` and `semantic_rank`.

4.2  BM25
    Library:      rank_bm25 (BM25Okapi)
    Fields:       title, keywords_clean, genres_clean, tagline, overview
    Weights:      title 1.0, overview 2.5, keywords 1.0, genres 1.0, tagline 0.5

    Overview gets the largest weight because the user is describing a plot,
    not a title.  Without this rebalance, a movie whose title literally
    contains "memory" or "stranded" would float to the top even when its
    overview has nothing to do with the user's intent.

4.3  RECIPROCAL RANK FUSION
    Constants:    RRF_K = 60, SEMANTIC_WEIGHT = 1.0, BM25_WEIGHT = 1.0
    Key:          fusion is keyed on `movie_key`, NOT on list position

    For every candidate in either list,
        contribution = SOURCE_WEIGHT / (RRF_K + rank + 1)
    Per-key contributions are summed; both source scores/ranks are
    preserved in the final dict so the benchmark and UI can see why a
    movie was promoted.

4.4  CROSS-ENCODER RERANKER
    Model:        BAAI/bge-reranker-v2-m3
    Pool:         RERANK_TOP_K = 30 candidates scored

    The document the reranker sees is built from
        Title — Year — Genres — Tagline — Overview (≤600 chars) — Keywords (≤200)
    in that order.  Ranking scores (RRF, semantic_score, vote_*) are NOT in
    the document, so the reranker cannot "leak" upstream signals into its
    own decision.

5. LLM COMPONENTS
================================================================================

All LLM work is gated through `src/llm/langchain_ollama.py`, which talks to
a local Ollama instance.  Every LLM call has:

    - a wall-clock timeout (`LLM_TIMEOUT_SECONDS`, default 25s)
    - a deterministic fallback that returns a sensible answer when the LLM
      times out, errors, or returns garbage.

This means the app never hard-fails because Ollama is down — it gracefully
degrades.  This was important in development on Windows where the Ollama
service occasionally hangs after a wake-from-sleep.

5.1  QUERY EXPANSION  —  `expand_query()`
    Returns a compact English retrieval query whose wording matches TMDB
    overview / keywords / tagline phrasing.  A prompt rule explicitly forbids
    adding movie titles unless the user named one — that protects against the
    LLM "helping" by injecting "Inception".

5.2  HyDE  —  `hyde_generate()`
    Returns a short synthetic English overview (2–3 sentences) for the user's
    request.  We embed THIS string and use it for a second semantic-retrieval
    pass.  The two retrievals are merged with RRF — rank position matters,
    not raw similarity score, because the two queries are embedded against
    different reference strings and their absolute scores aren't comparable.

5.3  EXPLANATIONS  —  `explain_movie()` / `explain_movies_batch()`
    The contract is structured JSON:
        {"evidence": "<verbatim phrase from supplied metadata>",
         "match_strength": "strong|moderate|weak",
         "explanation": "..."}
    On every reply, `_accept_grounded_item()` checks that the `evidence`
    string is LITERALLY present in the supplied metadata (lowercased
    substring match).  If the LLM claims evidence that isn't actually in the
    metadata — fabrication — we throw the reply away and use the
    deterministic fallback.

    The deterministic fallback never invents content.  It states which
    metadata fields are available and which user-query concepts overlap
    them (using a small synonym map: dream/dreams/nightmare/lucid,
    reality/realities/alternate, heist/thief/steal/espionage, etc.).  If
    nothing overlaps, it says so plainly.

5.4  WHY EVIDENCE VERIFICATION MATTERS
    Without it, the LLM tends to write fluent-sounding "Inception fits
    your dream-heist query because of its mind-bending plot" prose for
    every result, even for movies that don't fit at all.  With the
    verifier, every visible explanation is anchored to a real phrase
    in the actual metadata; if the model wants to praise something
    it must point at where that praise lives.

6. THE THREE PIPELINES
================================================================================

6.1  BASIC  —  `src/pipelines/basic.py`
    Flow:
        normalize → parse_filters → semantic_search → dedup → top K
    Use case:  fastest, no LLM required, useful as a baseline.

6.2  ADVANCED  —  `src/pipelines/advanced.py`
    Flow:
        normalize
            → LLM expand_query
            → (optional) hyde_generate
            → semantic(expanded)
            → if HyDE: semantic(hyde) then RRF the two semantic rankings
            → dedup
            → cross-encoder rerank (30 candidates)
            → dedup
            → top K
            → batched grounded LLM explanation for top EXPLAIN_TOP_K
    Use case:  best for vague or paraphrased queries; the LLM expansion
    rewrites the user's request into TMDB-style retrieval text, and HyDE
    provides a second viewpoint that helps when the expanded keyword form
    still doesn't quite reach the target movie.

6.3  HYBRID  —  `src/pipelines/hybrid.py`
    Flow:
        normalize
            → LLM expand_query  (Phase 3 — gated on HYBRID_USE_LLM_EXPANSION)
            → semantic(expanded) + bm25(expanded)
            → dedup each
            → rrf_fusion
            → dedup
            → cross-encoder rerank (30 candidates)
            → dedup
            → top K
            → batched grounded LLM explanation for top EXPLAIN_TOP_K
    Use case:  best when literal-keyword evidence matters; BM25 catches
    movies whose overview contains the user's exact words, while semantic
    catches movies whose plot meaning aligns even without word overlap.

7. QUALITY ENGINEERING — WHY THINGS ARE DONE THIS WAY
================================================================================

7.1  ENGLISH-ONLY SCOPE
    Earlier versions of the project had a Vietnamese-to-English phrase map
    glued onto the query processor.  The map was noisy and only covered a
    handful of cases.  Worse, the TMDB index is English, so a translated
    Vietnamese query was still embedded against English content, and the
    translation often hurt retrieval more than it helped.  Removing it made
    every metric improve.

7.2  EVIDENCE-GROUNDED EXPLANATIONS
    LLMs love to bluff.  Asking "why does this movie match?" reliably gets
    plausible-sounding praise that does not stand up to a sniff test against
    the overview.  The evidence verifier turns the prompt into a checkable
    contract: the LLM must point at a verbatim phrase, and we check that
    phrase exists in the metadata before we show the explanation.  Failed
    checks fall back to a deterministic explanation that never lies.

7.3  TITLE-WEIGHT REBALANCE IN BM25
    Field-aware BM25 with overview-weighted-highest replaced an earlier
    title-dominant scheme.  Side effect: movies whose title literally
    contains a query token no longer reflexively win.  This is the right
    bias for natural-language descriptions.

7.4  HYBRID LLM EXPANSION ROUTED TO BOTH ARMS
    In Phase 3 we noticed Hybrid lost the "dream heist" case to "Dreamscape"
    because Inception's overview uses "subconscious" / "corporate espionage",
    not the literal word "dream".  The fix was to route the same expanded
    query Advanced uses into BOTH semantic and BM25 in Hybrid.  After the
    fix, Inception ranks #1 in Hybrid on that query.  The user's ORIGINAL
    query is still what the reranker sees, so stated-intent scoring stays
    canonical.

7.5  HyDE + RRF (NOT HyDE-AS-REPLACEMENT)
    Naively replacing the expanded query with the HyDE synthetic overview
    HURTS on benchmark cases where the LLM's synthetic prose drifts
    off-target (e.g. HyDE wrote "lucid dreaming has become a lucrative black
    market" for the dream-heist query; that wording matches Dreamscape better
    than Inception).  We instead run a second semantic retrieval against the
    HyDE prose and rank-fuse the two retrievals with RRF.  Rank position is
    fair across queries, raw similarity is not.

7.6  STABLE MOVIE KEYS + DEDUPLICATION EVERYWHERE
    `src/utils/dedup.get_movie_key()` returns a stable key from
    `(lower(title), year)` (with a tmdb_id fallback).  Every retrieval stage
    runs its output through `deduplicate_movies()`, and `find_duplicate_keys`
    surfaces survivors as warnings in the benchmark output.  Result: the UI
    never shows the same film twice, even if upstream retrievers surface
    multiple metadata rows for the same title.

7.7  DEBUG TIMING WITHOUT NOISE
    `DEBUG_RETRIEVAL` is `False` by default.  When set to `True`, the
    `timed()` context manager in `src/utils/debug.py` prints elapsed
    milliseconds for every stage in every pipeline.  When `False`, the same
    context manager does nothing — no overhead beyond a perf_counter call
    and a generator/try-finally.

8. CONFIGURATION KNOBS
================================================================================

All knobs live in `src/config.py`.  Important ones:

    EMBEDDING_MODEL           "BAAI/bge-m3"
    RERANKER_MODEL            "BAAI/bge-reranker-v2-m3"
    LLM_MODEL                 "llama3.2"   (or any local Ollama model)

    CANDIDATE_POOL            300          (wide semantic / BM25 first pass)
    RERANK_POOL               80
    RERANK_TOP_K              50           (how many candidates reach cross-encoder)
    FINAL_TOP_K               5            (UI output size)
    EXPLAIN_TOP_K             3            (how many results get an LLM explanation)

    BM25_TITLE_BOOST          1.0
    BM25_OVERVIEW_BOOST       2.5          (overview is what the user describes)
    BM25_GENRES_BOOST         1.0
    BM25_KEYWORDS_BOOST       1.0
    BM25_TAGLINE_BOOST        0.5

    RRF_K                     60
    SEMANTIC_WEIGHT           1.0
    BM25_WEIGHT               1.0          (lower to 0.8 only if BM25 literal hits
                                            dominate semantic intent post-expansion)

    LLM_TIMEOUT_SECONDS       25
    ENABLE_LLM_EXPLANATION    True
    HYBRID_USE_LLM_EXPANSION  True         (Phase 3)
    USE_HYDE_IN_ADVANCED      True         (Phase 6a)

    DEBUG_RETRIEVAL           False        (flip to True for stage timings)

9. BENCHMARK METHODOLOGY
================================================================================

`scripts/quality_smoke_test.py` runs ten canonical English queries through
each pipeline and reports, per (query × pipeline):

    - top-N result titles
    - whether the expected title appeared in top 3 / top 10
    - sem / bm25 / rrf / rerank / final scores
    - duplicate warnings (movie_key collisions, title+year collisions)
    - missing-score warnings
    - generic-or-unsupported explanation warnings (--no-llm skips this)

The ten cases are taken directly from `AGENTS.md` Task 8. The current
global-metadata build keeps films such as Leon: The Professional and
Parasite when English TMDB metadata is available, so those cases are now
normal in-scope benchmark hits.

Run it:
    python scripts/quality_smoke_test.py            # full benchmark, LLM on
    python scripts/quality_smoke_test.py --no-llm   # retrieval-only, fast
    python scripts/quality_smoke_test.py --modes Advanced
    python scripts/quality_smoke_test.py --queries "your custom query"

A lighter `_adhoc_benchmark.py` runs the same 10 cases without
explanations or score printing, for quick iteration during development.

10. RESULTS AND HONEST LIMITATIONS
================================================================================

10.1  HEADLINE NUMBERS (after accuracy optimization, global metadata build)

    Mode        top1   top3   top10   miss   exp_miss   / total
    Basic        8     10     10      0      0          / 10
    Advanced     10    10     10      0      0          / 10
    Hybrid       10    10     10      0      0          / 10

    Columns
        top1 / top3 / top10 — expected title appeared in top-1 / top-3 / top-10
        miss                — expected title absent on a case that SHOULD have hit
        exp_miss            — expected title absent because the case is marked
                              out of scope for this benchmark

    All ten in-scope cases land within top-10 across every mode, and the
    reranked modes land all ten expected titles at top-1.

10.2  WHAT EACH PHASE CHANGED

    Phase 1  (English metadata refactor, data rebuild)
        - Pre-rebuild Basic / Advanced / Hybrid: 3/5/8/2  7/8/8/2  6/7/8/2
        - Post-rebuild:                          6/8/8/2  7/8/8/2  6/7/8/2
        - Basic gained heavily from cleaner documents; Advanced and Hybrid
          held steady at the top.

    Phase 2  (benchmark rewrite)
        - No accuracy change.  Added expected-title checks and the
          exp_miss column so we stop counting explicit out-of-scope cases as
          regressions.

    Phase 3  (Hybrid uses LLM expansion for both semantic and BM25)
        - Hybrid 5/8/8/2 → 6/8/8/2.
        - The "dream heist" case: Hybrid was ranking Dreamscape #1 over
          Inception because Inception's overview uses "subconscious", not
          the literal "dream".  After Phase 3 the expanded query reaches
          BM25, Inception gets a BM25 contribution, and the reranker
          places it #1.

    Phase 5  (DEBUG_RETRIEVAL stage timings)
        - No accuracy change.  Adds opt-in millisecond timings for every
          pipeline stage.

    Phase 6a  (HyDE in Advanced + RRF + wider candidate pool)
        - Advanced 6/8/8/2 (parity).  No regression, no measurable gain
          on the 10 canonical cases.  Infrastructure now exists for
          queries where the user's wording is far from any TMDB overview.

    Accuracy optimization pass  (global metadata + deterministic expansion)
        - Removed the original-language-only ingest restriction while keeping
          English metadata as the query/index language.
        - Rebuilt the CSV and Chroma index: DATASET_ROW_COUNT now verifies
          against the live row count.
        - Added deterministic metadata expansion and calibrated reranker
          priors. Final benchmark: Basic 8/10 top-1, Advanced 10/10 top-1,
          Hybrid 10/10 top-1, zero misses.

10.3  LIMITATIONS WE CAN NAME HONESTLY

    a) The benchmark is small (10 cases).  Numbers are illustrative, not
       statistically significant.

    b) Deterministic query expansion improves known metadata-language gaps
       but it is still a hand-authored rule set. A larger benchmark should
       be used before adding many more domain aliases.

    c) HyDE adds an extra LLM call (~3-5 s) to every Advanced query.  It
       does not improve the 10-case benchmark.  It is kept on by default
       because it adds genuine recall headroom for harder queries, but a
       latency-sensitive deployment should set `USE_HYDE_IN_ADVANCED = False`.

12. FUTURE WORK
================================================================================

Multilingual retrieval
    Add multilingual query handling (for example Vietnamese-to-English
    translation before retrieval, or a dual-language index). The current
    data build can include global films, but the query language is still
    expected to be English.

Reranker quality on aesthetic-tone queries
    The calibrated reranker works on the smoke benchmark, but a larger
    benchmark should test whether the vote-count prior is still optimal
    across obscure-title searches, recent releases, and niche genres.

True multi-query RRF
    Phase 6a fuses two semantic retrievals (expanded + HyDE).  A
    deeper version would generate N paraphrases of the user query,
    run N retrievals, and RRF all of them.  Expected to help most on
    short or vague inputs.

User profiles and history
    The current system is stateless.  Adding watch history, ratings,
    or a taste profile would let the recommender personalise — but
    only if added transparently, with a clear opt-in.

Benchmark expansion
    Ten cases is illustrative; a hundred would be statistical.  Adding
    cases for genre intent ("a cosy mystery in a small town"), tone
    intent ("something hopeful after a bad day"), and edge cases
    (single-word queries) would tell us which of the current
    components is the actual bottleneck.
