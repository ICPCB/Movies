# Track A — RG-03: q07 targeted label regrade

## Finding that changes this ticket — the regrade flow is hardcoded

The QL-01 ask was "use the existing RG-style flow: `build_regrade_sheet` →
human review → `check_regrade_sheet` → `merge_labels`." Inspection shows the
flow **cannot regrade q07 as-is**:

- `build_regrade_sheet.py` is hardcoded to `REVIEW_QIDS = ("q12", "q13")` and
  `RETRIEVAL_MISS_QIDS = ("q03", "q08")`. It builds exactly those rows and
  refuses to rebuild when `regrade_sheet.jsonl` exists.
- `check_regrade_sheet.py` reconstructs the *expected* rows from that same
  hardcoded logic and rejects any sheet whose rows/qids differ from the
  manifest.
- The existing `regrade_sheet.jsonl` already holds **45 completed human gold
  grades** (q03/q08/q12/q13). It must not be destroyed.

So Track A is **not** "run four commands." It requires a small, careful
**eval-harness** extension (not `src/*`) before the human regrade can happen.
`merge_labels.py` itself needs **no change** — it reads `rows_total` from the
manifest and its merge is qid-agnostic.

## Goal

Add q07 to the human regrade set so the q07 eval grades reflect reality —
specifically so a human re-grades What We Do in the Shadows (2014, tmdb 246741)
and My Babysitter's a Vampire (tmdb 63700), plus the rest of the q07 top-5
union for internal consistency — then re-merge authoritative labels and
metrics. No `src/*` change.

## Phases

- **A1 — tooling extension (Codex).** Add an additive q07 batch (batch 3,
  `batch_purpose: "ql_01_label_followup"`) to the regrade flow. The q07 batch
  is built like batch 2 (top-5 union across modes from
  `analysis/error_report/per_query_mode.jsonl`) but for `qid == "q07"`. The
  build must be **additive and non-destructive**: existing batch-1/batch-2 rows
  and their filled `gold_grade`/`gold_notes` must be byte-identical afterward.
  `check_regrade_sheet.py` reconstruction and `PREFERRED_QID_ORDER` are updated
  in lockstep. Ships with tests.
- **A2 — human regrade (HUMAN, external).** A human fills `gold_grade` /
  `gold_notes` for the new q07 batch-3 rows only. Expected outcome to verify,
  not assume: What We Do in the Shadows (2014) is the strongest match for the
  q07 query; My Babysitter's a Vampire is not a mockumentary.
- **A3 — check + merge (Codex).** Run `check_regrade_sheet` (must report
  `complete: true`), then `merge_labels`, refreshing `gold_labels.jsonl` and
  `metrics.json`. Record the q07 grade changes and the metric delta in the
  RG-03 report.

# Track B — DECOMP-01: decomposition-enriched eval for q05 and q10

## Finding that shapes this ticket — the decomposition does not exist

HY-FIX repeatedly stopped on "the artifact lacks full-pool decomposition."
That decomposition genuinely does not exist and cannot be derived from current
artifacts:

- `candidates.jsonl` stores per-stage scores but only for the ~10-15-row
  per-query top-k **union** — not the full rerank pool.
- The HY-STAB-01 / HY-FIX-02A traces captured the **target plus its RRF
  neighborhood**, not every pool member (HY-FIX-02B: "trace lacks full source
  rank lists").

So DECOMP-01 must **re-run the deterministic hybrid arms** (`pinned`, `no_llm`)
for q05 and q10 and dump the full pool. That is a **model-backed run**
(BGE-M3 embeddings + `bge-reranker-v2-m3` cross-encoder, GPU). It runs `src`
read-only via the eval harness; it does **not** edit `src/*`.

## Goal

For q05 (Thanatomorphose) and q10 ([REC]), produce a full rerank-pool and
final-blend score decomposition that lets an analyst **prove or rule out** a
safe localized Phase 5 fix — i.e. answer the question HY-FIX could not: does a
bounded cutoff or final-blend reweight pull the target into the top 5
**without** broadly re-ordering other results?

## Method

A new analysis script re-runs the `pinned` and `no_llm` deterministic arms for
q05/q10 (reusing the HY-STAB-01 `hybrid_expansion_stability.py` arm machinery
and `_decompose_pool`, and the HY-FIX-02A `rrf_pool_trace.py` pool logic). For
each arm it dumps, for **every member of an extended rerank pool** (at least
through the target's RRF rank — q05 pinned RRF 66, q10 pinned RRF 53 — so the
target is always captured): `semantic_score`, `bm25_score`, `rrf_score`,
`rerank_score`, `final_score`, and the final-blend formula inputs and weights.
From that it computes, per candidate localized policy (rerank cutoff increase;
final-blend reweight): (a) does the target reach top 5; (b) the **collateral**
— how many non-target pool members change rank, and by how much. A policy is
"safe" only if it rescues the target with bounded, quantified collateral.

## Files that DECOMP-01 would create (Codex stops here per the QL-01 ask — see note)

- `eval/scripts/decomp_pool_q05_q10.py` — model-backed decomposition runner +
  policy/collateral analysis.
- `eval/tests/test_decomp_pool_q05_q10.py` — unit tests (pure functions:
  decomposition aggregation, policy rescue check, collateral count — tested on
  fixtures, offline).
- `eval/runs/2026-05-19-1846-nogit/analysis/decomp/q05_q10_pool_decomposition.json`
  — the decomposition + policy/collateral artifact.
- `docs/superpowers/reports/decomp-01-q05-q10.md` — the prove/rule-out report.
- `docs/superpowers/AUTONOMOUS_CHECKPOINT_LEDGER.md` — append DECOMP-01
  checkpoints.

No `src/*` file is created or modified. A Phase 5 `src/*` change is **not** in
DECOMP-01's scope — it is gated behind DECOMP-01's decision (see Gate).

## Files to read but not change

- `eval/scripts/hybrid_expansion_stability.py` — deterministic-arm machinery,
  `_decompose_pool`.
- `eval/scripts/hy_fix_rrf_pool_trace.py` — RRF-pool trace logic to extend.
- `eval/scripts/_run_io.py`, `eval/scripts/_schemas.py`.

## Files forbidden

`src/**` (no edits) · `eval/queries/*` · any `*_labels.jsonl` · q07 anything

## Acceptance criteria

1. `decomp_pool_q05_q10.py` imports `src` only to run the existing pipeline;
   it adds **no new LLM call** inside retrieval/BM25/RRF/reranker code and
   makes **no `src/*` edit**.
2. The artifact contains, for q05 and q10, for both deterministic arms, a
   full extended-pool decomposition with all five stage scores per member and
   the final-blend formula.
3. For each candidate localized policy the artifact records: target rescued

   non-target rank changes).
4. The artifact carries an explicit `decision`: `safe_localized_fix_proven`
   (with the exact bounded change and its allowed `src/*` file) **or**
   `safe_localized_fix_ruled_out` (with the reason).
5. Unit tests pass offline; the model-backed run is recorded with its actual
   cost/time; `git diff --name-only -- src/` is empty.
